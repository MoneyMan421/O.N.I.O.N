## Change Categories

We use the following categories to organize changes:

- **Added** - New features or capabilities
- **Changed** - Changes to existing functionality
- **Deprecated** - Features that will be removed in future versions
- **Removed** - Features that have been removed
- **Fixed** - Bug fixes
- **Security** - Security improvements or vulnerability fixes

## Versioning Strategy

O.N.I.O.N follows semantic versioning:

- **MAJOR** version: Incompatible API changes or architectural changes
- **MINOR** version: New functionality in a backwards-compatible manner
- **PATCH** version: Backwards-compatible bug fixes

## Layer-Specific Changes

Changes are tracked by which layers of the O.N.I.O.N architecture they affect:

- **L1 - Entry/Trigger**: Source verification, GitHub Actions
- **L2 - Build**: Build processes, artifact creation
- **L3 - Test**: Testing frameworks, quality gates
- **L4 - Policy**: Policy decision point, compliance rules
- **L5 - Control**: Approval gates, human oversight
- **L6 - Deploy**: Deployment processes, runtime enforcement
- **L7 - Runtime**: Runtime verification, health checks
- **L8 - Audit**: Audit logging, evidence collection
- **L9 - Monitor**: Monitoring, feedback loops, anomaly detection
